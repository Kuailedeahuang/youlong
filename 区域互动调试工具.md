# 游戏内交互区域调试面板技术设计文档

> 版本：v1.0  
> 目标：在微信小游戏中集成可视化调试面板，支持交互区域的实时编辑（框选、文字、事件绑定），且**不影响正常游戏逻辑与点击交互**。

---

## 一、设计目标

| 目标 | 说明 |
|------|------|
| **非侵入式** | 调试面板代码与正式游戏逻辑解耦，发布时可通过编译开关移除 |
| **运行时编辑** | 不重启游戏即可增删改交互区域，实时看到效果 |
| **零性能损耗** | 非调试模式下，无任何额外代码执行（包括事件监听） |
| **操作隔离** | 调试面板的触摸事件不干扰原有的点击检测系统 |
| **配置持久化** | 编辑后的区域配置可保存至本地存储，并支持导出/导入 |

## 三、模块设计

### 3.1 调试模式开关

**功能**：通过隐蔽手势（如三指长按3秒）或编译常量开启/关闭调试面板。

**实现要点**：
- 使用 `wx.onTouchStart` / `wx.onTouchEnd` 检测特殊手势，**独立于 UIManager 的点击系统**。
- 手势检测器只在调试构建中存在，发布版完全剔除（通过构建工具条件编译）。
- 开启后显示调试面板UI，并**暂时禁用原有交互区域的点击响应**（避免冲突）。

### 3.2 调试面板UI

**形态**：半透明浮动面板，悬浮在游戏Canvas上方（使用绝对定位的 `div` 或 `canvas` 绘制）。

**组成部分**：

| 组件 | 功能 |
|------|------|
| 工具栏 | 新增区域、保存配置、导入/导出、退出调试模式 |
| 区域列表 | 显示所有区域ID，支持选中、删除、复制 |
| 属性编辑器 | 修改id、label、icon、actionName、样式参数 |
| 画布覆盖层 | 在高亮显示所有区域矩形（拖拽移动/缩放），与游戏Canvas同尺寸但透明 |

**实现方式**：
- 推荐使用 `div` + `SVG` 或 `Canvas` 作为覆盖层，避免污染游戏主Canvas的绘制状态。
- 覆盖层捕捉所有触摸/鼠标事件（通过 `pointer-events: auto`），原游戏Canvas设置 `pointer-events: none`（仅在调试模式下）。
- 覆盖层上的操作（拖拽矩形、点击按钮）不传递到游戏层。

### 3.3 区域编辑器

**核心能力**：
- 从 `InteractiveArea` 管理器加载当前区域配置，在覆盖层按**像素坐标**显示可拖拽矩形。
- 拖拽移动矩形 → 实时更新百分比坐标。
- 拖拽矩形四角 → 实时更新宽高百分比。
- 双击矩形打开属性面板。

**技术选型**：
- 不引入 Fabric.js（避免包体增大），使用原生 Canvas + 鼠标事件实现简单矩形编辑器。
- 或者使用 `SVG` + `svg.js` 轻量库，体积更小。

**坐标同步**：
- 游戏Canvas尺寸变化（如旋转屏幕）时，重新计算所有矩形的像素位置。
- 使用 `ResizeObserver` 监听游戏Canvas尺寸变化。

### 3.4 配置存储与导出

**存储**：`wx.setStorageSync('debug_areas', areasConfig)` ，不影响正式配置。

**导出**：提供“复制JSON”按钮，将配置拷贝到剪贴板，可粘贴到项目代码中替换硬编码。

**导入**：支持粘贴JSON配置，覆盖当前区域。

**重置**：恢复到从游戏原始配置（代码中硬编码的默认区域）。

### 3.5 与原有系统的隔离策略

为保证“不对当前除点击交互的部分产生影响”，必须做到：

| 隔离点 | 措施 |
|--------|------|
| **代码隔离** | 调试面板全部代码置于 `debug/` 目录，通过构建工具（如Webpack的 `DefinePlugin`）条件编译，发布版本不打包 |
| **事件隔离** | 调试模式下，通过设置一个全局标志 `window.__DEBUG_MODE__`，UIManager 在检测点击时如果该标志为 `true`，则**不触发原有 `addButton` 的回调**，将事件交由调试面板处理 |
| **绘制隔离** | 调试面板独立绘制在高层次覆盖层，不修改游戏主Canvas的渲染逻辑 |
| **配置隔离** | 调试面板修改的配置保存在 `wx.setStorageSync('debug_override_areas')`，游戏启动时检查该存储，若有则优先使用（仅供调试），删除存储后恢复原始配置 |

**关键代码示例（UIManager改造）**：

```javascript
// UIManager.js 中点击检测部分
handleTap(e) {
    if (window.__DEBUG_MODE__) {
        // 将事件传递给调试面板，不执行原有交互
        DebugPanel.getInstance().handleEvent(e);
        return;
    }
    // 原有逻辑
    this.checkButtons(e);
}
```

## 四、数据结构

沿用原有的 `InteractiveAreaConfig`，增加一个内部字段 `_debugSelected`（仅调试使用）。

// 调试面板内部扩展的配置（存储时过滤掉）

    ...InteractiveAreaConfig,

## 五、交互流程

### 5.1 开启调试模式

```
用户三指长按3秒
    ↓
手势检测器触发
    ↓
设置 window.__DEBUG_MODE__ = true
    ↓
UIManager 下次点击时切换到调试分支
    ↓
创建并显示调试面板UI（覆盖层）
    ↓
从存储或原始配置加载现有区域 → 绘制可编辑矩形
```

### 5.2 编辑区域

```
用户拖拽矩形边缘
    ↓
覆盖层鼠标事件 → 计算新像素尺寸
    ↓
转换为百分比坐标 → 更新内部配置
    ↓
实时重绘覆盖层矩形
```

### 5.3 保存配置

```
用户点击“保存”按钮
    ↓
将当前区域配置（去除 _debug 字段）保存至 wx.setStorageSync('debug_override_areas')
    ↓
同时更新 InteractiveArea 管理器中的配置（使游戏内立即生效）
    ↓
提示保存成功
```

### 5.4 退出调试模式

```
用户点击“退出调试”或再次长按3秒
    ↓
销毁覆盖层UI
    ↓
设置 window.__DEBUG_MODE__ = false
    ↓
UIManager恢复正常点击响应
    ↓
（可选）清除存储中的调试配置，恢复原始配置
```

## 六、接口定义

### 6.1 DebugPanel 单例类

```javascript
class DebugPanel {
    static getInstance();
    show();                       // 显示面板
    hide();                       // 隐藏面板
    loadAreas(areas);             // 从配置加载区域
    saveAreas();                  // 保存到存储与运行时管理器
    exportConfig();               // 返回JSON字符串
    importConfig(jsonStr);        // 导入JSON
    resetToOriginal();            // 重置到原始配置
    handleEvent(touchEvent);      // 处理触摸事件（由UIManager传递）
}
```

### 6.2 构建条件编译

在 `game.js` 入口：

```javascript
// 通过构建工具注入 __DEV__ 常量
if (__DEV__) {
    require('./debug/DebugPanel').init();
}
```

Webpack 配置示例：

```javascript
new webpack.DefinePlugin({
    __DEV__: JSON.stringify(process.env.NODE_ENV === 'development')
})
```

## 九、注意事项

1. **微信小游戏环境限制**：没有真正的DOM，所有UI需用Canvas绘制或使用 `wx.createCanvas` 创建覆盖层，事件处理使用 `wx.onTouchStart` 等，注意事件冒泡和阻止传递。
2. **覆盖层性能**：覆盖层每帧重绘开销较小，但避免频繁全屏重绘，只在矩形变化时重绘。
3. **配置冲突**：调试模式下的区域配置应完全替代原始配置，退出时建议提示用户是否保存更改。
4. **手势冲突**：三指长按可能与游戏内已有的多点触控手势冲突，可通过仅在未触摸UI区域时检测降低干扰。
5. **发布版清除**：务必使用条件编译，确保调试代码不会进入生产版本，避免安全风险和性能损耗。

## 十、总结

该设计通过引入**独立覆盖层 + 事件分流 + 条件编译**，实现了一个完全非侵入式的游戏内调试面板。AI开发人员可依据上述模块划分和接口定义，逐步生成完整的代码实现。核心难点在于小游戏环境下的覆盖层事件管理，但通过统一事件分发可以圆满解决。
