## P0 级问题

### 问题 1: 健身房小游戏 - 指示条完全不动

**相关文件**: 
- `js/main.js:126`
- `js/miniGames/GymMiniGame.js:82`

**问题描述**:
健身房小游戏启动后，指示条完全看不到移动，玩家无法进行游戏。

**根本原因 - deltaTime 单位不一致**:

```javascript
// main.js:126 - deltaTime 单位是秒
this.deltaTime = (now - this.lastTime) / 1000

// GymMiniGame.js:82 - 期望 deltaTime 是毫秒
const moveAmount = this.barSpeed * (deltaTime / 16.67)
```

**计算验证**:
```
假设 deltaTime = 0.01667 秒 (60fps)
moveAmount = 1.8 * (0.01667 / 16.67) = 1.8 * 0.001 = 0.0018

trackWidth ≈ 屏幕宽度 * 0.72 ≈ 432 (假设屏幕600宽)

移动完整条需要: 432 / 0.0018 = 240,000 帧
240,000 帧 / 60fps = 4,000 秒 ≈ 66分钟
```

**结论**: 指示条需要 66 分钟才能走完整个轨道，实际上完全不可用。

### 问题 2: 工作小游戏 - 模态框未关闭导致重叠显示

**相关文件**:
- `js/systems/WorkSystem.js:64-88`
- `js/scenes/sceneWithBackground.js:234-238`
- `js/core/uiManager.js:9-11`

**问题描述**:
点击"上班"或"加班"后，小游戏界面显示了，但之前的"工作信息"模态框没有消失，两者重叠显示。

**完整证据链**:

```
1. 用户点击"上班"按钮
   ↓
2. WorkSystem.showWorkModal() 打开模态框
   ├── type: 'action'
   ├── title: '工作 - xxx'
   └── actions: [{ text: '上班', callback: () => this._startWorkWithMiniGame(...) }]
   ↓
3. 用户点击"上班"按钮 → 触发 _startWorkWithMiniGame()
   ↓
4. _startWorkWithMiniGame() 没有关闭之前的模态框 ❌
   ↓
5. sceneWithBackground.render() 中 this.game.uiManager.clear() 
   └── clear() 只清除了按钮，没有清除 modals 数组！❌
   ↓
6. uiManager.modals 数组中仍有旧模态框
   ↓
7. 最终渲染时旧模态框仍显示在新游戏之上
```

**根本原因**:

| 位置 | 问题 |
|------|------|
| `WorkSystem._startWorkWithMiniGame()` | 没有调用 `uiManager.closeModal()` 或 `closeAllModals()` |
| `UIManager.clear()` | 只清 `pendingButtons`，不清 `modals` |

### 问题 3: 健身房系统 - 连续健身天数每次都加精力上限

**相关文件**: `js/systems/GymSystem.js:72-76`

**问题代码**:
```javascript
state.consecutiveGymDays++

if (state.consecutiveGymDays >= 3) {
    state.maxEnergy = Math.min(15, state.maxEnergy + 1)
}
```

**问题描述**:
- 每次健身都会增加 `consecutiveGymDays`
- 判断条件是 `>= 3`，意味着**第3天及之后的每一次健身都会增加精力上限**
- 预期逻辑应该是：每连续健身3天奖励一次，然后重置计数

**复现路径**:
```
第3天健身 → consecutiveGymDays = 3 → maxEnergy +1 ✅ (正确)
第4天健身 → consecutiveGymDays = 4 → maxEnergy +1 ❌ (不应该再加)
第5天健身 → consecutiveGymDays = 5 → maxEnergy +1 ❌ (不应该再加)
...
玩家可以无限增加精力上限到15
```

### 问题 4: 健身房系统 - 连续健身天数跨天不重置

**相关文件**: 
- `js/core/gameState.js:612-617`
- `js/managers/backGameManager.js:215`

**问题代码**:
```javascript
// gameState.js nextDay() 方法
nextDay() {
    this.data.day++
    this.data.energy = this.data.maxEnergy
    this.data.newspaperShown = false
    this.data.marketEnteredToday = false
    this.data.todayEvents = []
    // ... 没有 consecutiveGymDays 的重置逻辑！❌

```

**问题描述**:
- `nextDay()` 方法中**没有重置 `consecutiveGymDays`**
- 只有在 `doEntertainment()` (娱乐) 中才会重置
- 这意味着：如果玩家某天不健身但也不娱乐，连续健身天数**不会重置**

**复现路径**:
```
第1天健身 → consecutiveGymDays = 1
第2天不健身也不娱乐 → consecutiveGymDays 仍然是 1 ❌
第3天健身 → consecutiveGymDays = 2 (应该是1！)
```

### 问题 5: 小游戏期间场景切换导致状态丢失

**相关文件**: `js/scenes/sceneWithBackground.js:150-155`

**问题代码**:
```javascript
async onEnter(params) {
    if (this.activeMiniGame) {
        this.activeMiniGame = null  // 直接清空，没有调用 onComplete ❌
    }
    // ...

**问题描述**:

5. 回调未执行，玩家损失精力但没获得工资 ❌
```

## P1 级问题

### 问题 6: 健身小游戏 - 位置判定使用相对坐标

**相关文件**: `js/miniGames/GymMiniGame.js:285-299`

**问题代码**:
```javascript
handleTouch(x, y) {
    const trackY = this.game.renderer.height * 0.38
    const touchRange = 50
    
    if (y >= trackY - touchRange && y <= trackY + 72 + touchRange) {
        const barCenter = this.barX  // ❌ 这是相对于trackX的偏移量
        
        if (barCenter >= this.sweetZoneStart && barCenter <= this.sweetZoneEnd) {
            result = 'perfect'
        }
    }
}
```

**问题分析**:
- `barX` 是指示条在**轨道内的相对偏移**（范围 0 到 `trackWidth`）
- `sweetZoneStart/sweetZoneEnd` 也是相对于 `trackX` 的偏移量
- **但触摸坐标 `(x, y)` 是屏幕绝对坐标**
- 正确做法应该是：`const barCenter = this.trackX + this.barX`
- 当前代码直接将相对偏移 `barX` 与绝对触摸坐标 `x` 比较，导致**几乎不可能命中正确位置**

### 问题 9: 工作小游戏 - 目标时间分配不合理

**相关文件**: 
- `js/data/gameConfig.js:118-127`
- `js/miniGames/WorkMiniGame.js:44-56`

**配置参数**:
```javascript
work: {
    timeLimit: 5000,        // 游戏总时长 5秒
    totalTargets: 8,        // 8个目标
    targetLifetime: 1500,   // 目标存活时间 1.5秒
}
```

**计算分析**:
```
目标生成间隔 = 5000 / 8 = 625ms
目标出现时间: 0, 625, 1250, 1875, 2500, 3125, 3750, 4375ms
目标过期时间: 1500, 2125, 2750, 3375, 4000, 4625, 5250, 5875ms
```

**问题描述**:

| 目标 | 出现时间 | 过期时间 | 游戏结束时间 | 实际可点击时间 |
|------|----------|----------|--------------|----------------|
| #7 | 3750ms | 5250ms | 5000ms | **1250ms** (少于1500ms) |
| #8 | 4375ms | 5875ms | 5000ms | **625ms** (少于1500ms的一半!) |

- **第7、8个目标的有效存活时间被游戏结束时间截断**
- 玩家对最后两个目标的反应时间严重不足

### 问题 10: 工作小游戏 - 时间计算使用 Date.now()

**相关文件**: `js/miniGames/WorkMiniGame.js:59-62`

**问题代码**:
```javascript
update(deltaTime) {
    if (!this.active) return
    this.elapsed = Date.now() - this.startTime  // ❌ 应该用累加 deltaTime
}
```

**问题分析**:
1. 配置文件设置 `timeLimit: 5000` (5秒)
2. 当前使用 `Date.now() - startTime` 计算已用时间
3. **问题**: 当小游戏运行时弹出模态框，触摸事件被 `UIManager` 拦截
4. 模态框打开后，游戏继续运行（`update` 循环不停止），`elapsed` 持续增加
5. 导致玩家**无法操作时游戏仍在倒计时**

**同样的问题存在于**: `js/miniGames/GymMiniGame.js:76-79`

## 设计缺陷总结

### 1. 模态框管理缺陷

```
设计缺陷: 场景系统假设一个时刻只显示一种UI
实际情况: 小游戏需要关闭所有UI，但系统没有提供统一关闭机制

应该有的流程:
开始工作 → 关闭"工作"模态框 → 启动小游戏 → 显示小游戏UI → 游戏结束 → 打开"结果"模态框
```

### 2. 时间单位不一致

```
main.js: 使用秒 (deltaTime 秒)
小游戏: 假设毫秒 (deltaTime / 16.67)

原因: 可能参考了其他平台（如Unity/Unreal）的实现，它们的一帧时间是可预测的
但浏览器 requestAnimationFrame 的 deltaTime 是变化的
```

### 3. UI层级管理混乱

```
UIManager.clear() vs UIManager.closeAllModals()
render() 中 modals 独立渲染，不受 clear() 影响
sceneWithBackground.render() 中的 clear() 无法清除模态框
```

### 4. 连续健身机制设计缺陷

```
预期: 连续健身3天奖励一次精力上限，然后重置计数
实际: 
1. 每次健身(第3天起)都增加精力上限
2. 跨天不重置计数，只有娱乐才重置
3. 导致连续健身机制完全失效
```

### 5. 小游戏生命周期管理缺陷

```
问题: 小游戏没有独立的生命周期管理
影响:
1. 场景切换时小游戏被强制终止，回调不执行
2. 模态框可能遮挡小游戏，导致无法操作
3. 没有暂停/恢复机制
```

## 修复建议（非强制）

### P0 高优先级修复

1. **统一 deltaTime 单位**
   - 方案A: main.js 传递毫秒单位的 deltaTime
   - 方案B: 所有小游戏改用累加 deltaTime 而不是 Date.now()

2. **小游戏启动时关闭所有UI**
